# 安否確認サービス サービス概要（チーム共有用 1枚もの）

> 詳細仕様は同フォルダ `requirements.md` を参照。本書は5分で全体像を掴むための入口資料。

---

## 1. ひとことで言うと

> まずこのサービスが何かを1文で説明します。

**災害発生時に、自社の社員（最大300人）に Amazon Connect から自動架電し、電話のダイヤルキー（DTMF）で安否を回答してもらうサーバーレスサービス。**

回答結果は管理サイトでリアルタイムに集計・可視化される。従来の電話連絡網運用の自動化版。

---

## 2. 主要ユースケース

> 誰が何をするかを、利用者ごとに下の表にまとめました。

| 利用者 | やること |
|---|---|
| **管理者** | 災害発生時に管理サイトから「安否確認サイクル」を手動で起動／進捗を見える化／履歴・録音を確認／社員マスタを管理（個別入力／CSV一括） |
| **一般社員** | セルフサービス画面で自分の連絡先を登録・更新／自分の過去応答履歴を閲覧／架電を受けてダイヤルキーで安否回答（1=無事 / 2=軽傷 / 3=重傷 / 4=出社不可） |

---

## 3. 全体フロー（典型シナリオ）

> 災害発生からサイクル完了までの一連の流れを、テキスト図で示します。

```
[災害発生]
    ↓
[管理者] 管理サイトにログイン → サイクル起動（対象範囲・再発信回数・間隔を指定）
    ↓
[Step Functions] サイクル実行を開始
    ↓
[Amazon Connect] 対象社員に1人ずつ自動架電
    ↓
[社員] ガイダンス聴取 → DTMFで1〜4を入力
    ↓
[Lambda] 応答をDynamoDBに記録、通話を録音してS3に保存
    ↓ (応答なし／無効入力なら Retry_Interval 経過後に再発信、最大 Retry_Count 回)
    ↓
[管理サイト] 10秒間隔ポーリングで進捗・集計をリアルタイム表示
    ↓
[完了] 全員の最終ステータス確定 → サイクル完了
```

SLA：**初回発信 30 分以内 / サイクル完了 60 分以内**（300人・再発信3回・間隔5分・同時10コール条件下）。

## 4. 主要コンポーネント（Glossary要約）

> サービスを構成する8つの論理コンポーネントと、その実体を下の表にまとめました。
>
> **要点：論理コンポーネント名と AWS サービスの対応関係です。**

| 名称 | 役割 | 実体 |
|---|---|---|
| **Admin_Console** | 管理者向けWeb管理サイト | CloudFront + S3 |
| **Employee_Self_Service** | 一般社員向けセルフサービス画面 | 同フロントエンドのロール別ビュー |
| **Auth_Service** | 認証・認可 | Amazon Cognito |
| **Cycle_Manager** | サイクルの起動・進捗管理 | Step Functions + Lambda |
| **Connect_Caller** | 自動架電・DTMF応答取得 | Amazon Connect + コンタクトフロー + Lambda |
| **Recording_Store** | 通話録音の保存・メタデータ管理 | S3 + Lambda |
| **Status_Viewer** | 進捗・結果のリアルタイム表示 | フロント＋API Gateway |
| **Employee_Master** | 社員マスタ | DynamoDB |

## 5. 使用 AWS サービス（NFR6）

> 使用する AWS サービスを列挙します。リージョン制約もここで明示します。

Route 53 / ACM / CloudFront / S3 / Cognito / **Amazon Connect** / **Step Functions** / Lambda / EventBridge / DynamoDB / API Gateway / KMS / CloudWatch Logs

リージョン：**東京（ap-northeast-1）固定**。マルチリージョンDRはスコープ外。

## 6. データストア構成

> DynamoDB 4テーブル＋S3 1バケットの構成を、下の表にまとめました。
>
> **要点：暗号化はすべて SSE-KMS。録音ファイル本体のみ S3、他は DynamoDB。**

| ID | 名称 | 種別 | キー設計 | 用途 |
|---|---|---|---|---|
| D1 | Employee_Master | DynamoDB | PK = 社員ID(UUID) | 社員マスタ（電話番号・所属・Cognito sub等） |
| D2 | Cycle | DynamoDB | PK = Cycle ID(UUID) | サイクル単位の起動情報・ステータス |
| D3 | Response | DynamoDB | PK = Cycle ID, SK = 社員ID | サイクル×社員の応答結果 |
| D4 | Recording_Metadata | DynamoDB | PK = Cycle ID, SK = 社員ID+通話Seq | 録音ファイルへの参照情報 |
| D5 | Recording_File | S3 | `recordings/{cycle_id}/{employee_id}/{seq}.wav` | 録音ファイル本体（90日で自動削除） |

**全データストアに SSE-KMS（カスタマー管理キー）適用。**

## 7. 主要制約・SLA

> 実装判断の前提になる数値・制約をすべて下の表に集約しました。実装中は何度も参照することになります。
>
> **要点：30分／60分／300人／同時10コール／99.9% の数値を覚えてください。**

| 項目 | 値 |
|---|---|
| 対象規模 | 最大 300 名 |
| 同時アクティブコール | 10（Amazon Connect 東京リージョン既定クォータ） |
| 初回発信完了 SLA | サイクル起動から **30 分以内** |
| サイクル完了 SLA | サイクル起動から **60 分以内** |
| Retry_Count（再発信回数） | 0〜5（既定値はParameter） |
| Retry_Interval（再発信間隔） | 1〜60 分 |
| DTMF応答待機 | ガイダンス再生後 10 秒（無効入力時は最大2回まで再入力促し） |
| 同時実行サイクル数 | **1 のみ**（実行中サイクルあれば新規起動は拒否） |
| 月間可用性目標 | **99.9%**（月間ダウンタイム 43.2 分以内）／東京リージョン内 2 AZ 以上 |
| 録音保管期間 | **90 日**（S3 ライフサイクルで自動削除） |
| CloudWatch Logs 保持期間 | 既定 90 日（Parameterで 1〜3653 日） |

## 8. セキュリティ要点

> 通信・認証・個人情報・録音・監査の5観点で要点を整理しました。

- **通信**：すべて TLS 1.2 以上（CloudFront・API Gateway・Cognito）
- **認証**：Cognito ユーザープール（管理者ロール／一般社員ロール）
  - 5 回連続失敗で 30 分ロックアウト
  - アクセストークン1時間 / リフレッシュトークン30日
- **個人情報（電話番号）**：
  - DynamoDB に SSE-KMS で暗号化保管
  - 監査ログ出力時は **末尾4桁以外を `*` でマスキング**
  - 一般社員は自分の電話番号のみ閲覧・編集可（他者要求は 403）
- **録音ファイル**：
  - S3 を SSE-KMS で暗号化
  - **再生は有効期限15分の署名付きURL経由のみ**
  - Lambda実行ロール／管理者Cognitoロール以外はバケットポリシー＋KMSポリシーで拒否
- **監査ログ**：認証・社員マスタ更新・サイクル起動・電話番号更新を CloudWatch Logs に出力

## 9. IaC・環境

> 環境構成の方針（単一CFnテンプレートで dev/stg/prod 切替）を箇条書きにしました。

- **単一の CloudFormation テンプレート**で `dev` / `stg` / `prod` を Parameter 切替
- Amazon Connect インスタンス ID と電話番号 ARN は Parameter で外部受領
- 東京リージョン以外・別アカウントへの展開はスコープ外（バリデーションエラー）

---

## 10. やらないこと（スコープ外）

> あえて作らない機能を明示します。誤実装防止のため必読です。

以下はあえて**作らない**。誤実装防止のため必読。

1. マルチリージョン DR、東京以外への展開
2. **SMS・メール・モバイルプッシュ通知**による安否確認（電話のみ）
3. **音声認識による自由発話応答**（DTMFキー入力のみ）
4. **Active Directory / Entra ID との SSO 連携**（Cognito スタンドアロン）
5. **自動トリガー起動**（緊急地震速報連携、EventBridge Scheduler等）— 管理者の手動起動のみ
6. 高度な監査機能（CloudTrail データイベント、Athena分析、S3 Object Lock 等）
7. 多言語ガイダンス、部署別レポート、訓練モード、外部者・委託社員への連絡、多拠点同期
